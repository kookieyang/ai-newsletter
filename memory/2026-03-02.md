### Bugs Fixed

1. **Multiline input disappearing** (main bug from yesterday)
   - Root cause: `input.style.height = 'auto'` collapsed textarea to 0 on every keystroke, hiding content
   - Fix: Use `1px` shrink + `requestAnimationFrame` resize after Enter/Backspace/Delete
   - Also: `flex: 1` on textarea was overriding explicit height → replaced with `flex-shrink: 0`

2. **Messages not rendering after code block**
   - Root cause: Message containing literal `<textarea>` injected as innerHTML, creating real DOM element that ate all subsequent content
   - Fix: Added `escapeHtml()` function to escape `&`, `<`, `>`, `"` before inserting

3. **Message polling dying silently**
   - Root cause: `oldScrollHeight` variable referenced but never declared → `ReferenceError` killed `setInterval` callback
   - Fix: Capture `oldScrollHeight` before DOM replace

4. **Copy button breaking on messages with quotes/backticks**
   - Root cause: `JSON.stringify(text)` in inline `onclick` handler broke HTML parsing
   - Fix: Store texts in JS `messageTextMap`, reference by ID

5. **Multiline messages losing newlines in history**
   - Root cause: Gateway strips `\n`, `U+2028`, zero-width chars from messages
   - Fix: Encode `\n` as ` ~~NL~~ ` ASCII marker before sending, decode on read
   - Invisible Unicode approaches all failed — gateway is aggressive about stripping

6. **Copy button including timestamp prefix and ~~NL~~ markers**
   - Fix: Strip `[Mon 2026-03-02 ...]` prefix and decode `~~NL~~` → `\n` in copy function

7. **Image attachments not being sent to AI**
   - Root cause: Server's `/api/send` only extracted `text`, ignored `attachments`
   - Fix: Convert data URLs to raw base64, pass as `{type, mimeType, fileName, content}` to gateway's `chat.send`

8. **Image thumbnails disappearing from message history**
   - Root cause: `parseMessages` only extracted `text` content blocks from user messages, ignored `image` blocks
   - Fix: Extract `type='image'` content blocks as attachments with data URLs

### Technical Notes

- Or find PID via `/usr/sbin/lsof -i :8899` then `kill -9 <PID>`

### Voice Chat — BlackHole Root Cause Found (17:30)
- Confirmed via mic-test.html: Chrome was using **BlackHole 2ch (Virtual)** as input device
- BlackHole is a silent virtual audio routing device — produces valid webm with zero audio data
- **Fix applied**: `startRecording()` now enumerates devices via `navigator.mediaDevices.enumerateDevices()` and skips devices matching `/blackhole|virtual|loopback|soundflower/i`, preferring real mics
- Falls back to default `{audio: true}` if no real mic found

### Voice Chat — Base64 Parsing Bug Fixed (17:33)
- Server log revealed: `base64len=94575, bytes=15` — data URL prefix wasn't stripped properly
- Root cause: regex `^data:[^;]+;base64,` fails on `data:audio/webm;codecs=opus;base64,` because `[^;]+` stops at first `;`
- Fix: changed to `^data:[^,]+,` which strips everything up to the comma
- Also fixed: `Cannot access 'audioBuffer' before initialization` — log line referenced variable before `const` declaration
- Also added `cleanBase64` stripping inside `transcribeAudio()` as defense-in-depth
- LaunchAgent restart flow: must `launchctl bootout` first, then kill, then start, then `launchctl bootstrap`
- **Status**: Fixes deployed, waiting for Louis to hard-refresh and test with correct mic device
